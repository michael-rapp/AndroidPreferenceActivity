1 Implementation details

The activity PreferenceActivity, which is provided by this library is an abstract class, which is extended from the Android SDK's class Activity. In order to use the library, the class PreferenceActivity simply has to be extended and the preference headers, which specify categories of preferences, must be added to the activity. Although it is possible to add the preference headers at runtime, the usual requirement is to initialize them once the activity is created. Therefore the class PreferenceActivity offers the protected method onCreatePreferenceHeaders():void, which amends the activity's lifecycle and is called after the onCreate-, but before the onStart-method in order to add the preference headers which should be shown as soon as the activity becomes visible. Multiple ways to override this method in order to add preference headers will be discussed in the following section "Adding and removing preference headers". The preference headers itself are represented by instances of the class PreferenceHeader. Such a class instance must at least specify a title, which is shown in the navigation to represent the preference header. Furthermore, a summary, which describes the preference header more detailed, an alternative title, which is shown when the preference header is currently selected, and an icon can be specified using the classes' setter methods. The preferences, which should be shown, when a preference header is selected by the user, must be contained by a fragment, which can be implemented by extending the Android SDK's class PreferenceFragment. Since version 1.1.0 the library does also provide the class PreferenceFragment, which is extended from the Android SDK's class to add the possibility to show a button, which allows to restore the default values of the fragment's preferences. Alternatively, every fragment, which is extended from the SDK's class Fragment and therefore must not necessarely contain preferences, can be shown by using the library. The fragment, which should be shown, when a specific preference header is selected, can be specified by passing its full qualified name to the appropriate setter method of the PreferenceHeader-instance. Besides, there is an other setter method, which allows to specify an intent, which is launched to start an activity once the preference header becomes selected. A fragment and an intent can be specified at the same time, but both of them can also be used on their own. The class PreferenceActivity does override the methods onSaveInstanceState(Bundle):void and onRestoreInstanceState(Bundle):void, which are called by the Android SDK during the activity's lifecycle in order to restore the activity's current state, when it is temporary destroyed. For this reason, the currently selected preference header will remain selected, even if the orientation of the device changes or if the activity is temporary brought to the background.

The activity PreferenceActivity has been designed in order to allow optical manipulations as easy as possible. Therefore the class offers various methods to access the activity's child views and to directly manipulate their most important properties (e.g. the width of the navigation on tablets). These methods, as well as the child views, which are used to realize the varying appearance on tablets and smartphones, will be discussed in the section "Manipulating the activity's appearance". Further manipulations, which take influence on the activity's behavior, will be discussed in the section "Changing the activity's behavior". To retrieve, whether the activity is run on a tablet and therefore the navigation and the currently selected fragment are shown split-screen, the method isSplitScreen():boolean can be used.

Furthermore, the activity can be used as wizard, which provides an alternative navigation, which allows to step from one preference header to an other, to the user. This possibility, which also allows to register listeners, that can be used for supervising the navigation and to validate the user's interactions, will be discussed in the last section "Using the activity as a wizard"

Since version 2.0.0 the UI, which is provided by the library's PreferenceActivity , is designed according to the Android 5.0 "Material Design" guidelines, even on pre-Lollipop devices (API lever less than 21). These guidelines can be found here. Using Material Design requires functionality, which has been introduced with the release of the Android SDK 21, such like a custom toolbar implementation, which replaces the ActionBar as used in Android 4 "Holo" styled apps. In order to be able to use this functionalities, since version 2.0.0 the library depends on the AppCompat v7 revision 21 support library. Furthermore, the theme, which is applied on a PreferenceActivity should be extended from the AppCompat library's theme Theme.AppCompat.NoActionBar or Theme.AppCompat.Light.NoActionBar, which includes no Holo-styled ActionBar, as the ActionBar is replaced by the library's built-in toolbar. When using the theme Theme.AppCompat.NoActionBar or Theme.AppCompat.Light.NoActionBar instead, the library's custom toolbar can not be used and a regular ActionBar will be shown instead. Anyways, as the library uses the AppCompat library it is indispensable to use one of the support library's AppCompat-themes. In the last section of this documentation further information about using the library in connection to themes will be discussed.

2 Adding and removing preference headers

In this section two ways of how to add preference headers to a PreferenceActivity will be discussed. The first possibility is to add the preference headers when the activity is created. This kind of implementation should be used when the preference headers should be initialized once and are not meant to be changed during the activity's lifetime. The second possibility is to add the preference headers at runtime. Such an implementation allows to dynamically add and remove preference headers during the activity's lifetime.

2.1 Adding preference headers when the activity is created

Like already mentioned, the preference headers, which should be shown as soon as a PreferenceActivity becomes visible, should be added by overriding the classes' onCreatePreferenceHeaders-method. The following methods are available to add preference headers to the activity:

addPreferenceHeader(PreferenceHeader):void

addAllPreferenceHeaders(Collection<PreferenceHeader>):void

addPreferenceHeadersFromResource(int):void

Unlike the first two methods mentioned above, the third one does not expect instances of the class PreferenceHeader as a parameter. Instead the resource ID of a XML resource, which specifies the preference headers, which should be added, must be passed to the addPreferenceHeadersFromResource-method. The syntax of the specified XML file, which should be present in the folder /res/xml, is similar to the syntax of the XML files, which can be used to specify preference headers for the Android SDK's built-in PreferenceActivity. An example of such a XML file can be found in the "Examples" section of this documentation.

In case the PreferenceHeader-instances, which should be added to the activity, should be created programmatically, the class offers the following setter methods to specify its properties. Besides, there are two constructors, which require to specify the preference header's title, which is shown in the activity's navigation to visualize the preference header. The first constructor does expect the title to be passed as an instance of the class CharSequence, the second one allows to pass the resource ID of a string resource together with the context, which allows to access the resource. These two possibilities are also provided by all setter methods, whose purpose is to specify a text property and which are listed below:

2.2 Adding preference headers at runtime

An alternative to overriding the onCreatePreferenceHeaders-method is to add the activity's preference headers at runtime. Both possibilities can also be combined to show some preference headers as soon as the activity becomes visible and afterwards add additional headers at runtime. For adding preference headers at runtime, the same methods as already mentioned above can be used. Additionally it is possible to remove preference headers, which have already been added to the activity, by using the methods listed below. The selected preference header will automatically adapted when a preference header is added to or removed from the activity. For example, if the currently selected preference header is removed, one of its neighbors will be selected.

removePreferenceHeader(PreferenceHeader):boolean

clearPreferenceHeaders():void

In case the preference headers are added and removed dynamically at runtime, it may be useful to retrieve some information about the preference headers, which currently belong to the activity. For this purpose the following methods are provided by the class PreferenceActivity:

getAllPreferenceHeaders():Collection<PreferenceHeader>

getNumberOfPreferenceHeaders():int

For even more control over the activity's preference headers, the adapter, which is used to organize the preference headers, can be referenced by using the method getListAdapter():PreferenceHeaderAdapter. This method returns an instance of the class PreferenceHeaderAdapter.

2.3 Programmatically selecting preference headers

The implementation of the class PreferenceActivity already takes care of selecting a preference header when it is clicked by the user and to show the according fragment, respectively to launch the the header's intent. To offer the possibility to programmatically select preference headers without any user interaction, the class PreferenceActivity provides the following methods. These methods allow to either pass a reference to the preference header, which should be selected, or to pass its position. Also for both possibilities an additional method, which allows to specify an instance of the class Bundle, exists. Such a bundle may be used to pass parameters to the fragment, which is shown, when the preference header is selected.

selectPreferenceHeader(PreferenceHeader):void

selectPreferenceHeader(PreferenceHeader, Bundle):void

selectPreferenceHeader(int):void

selectPreferenceHeader(int, Bundle):void

The selection model of the activity's navigation is managed by a ListView, which can be referenced by using the method getListView():ListView. This allows to retrieve additional information about the navigation's current state. Alternatively, the class PreferenceActivity offers the following methods to retrieve the information about the currently selected preference header:

isPreferenceHeaderSelected():boolean

getSelectedPreferenceHeader():PreferenceHeader

getSelectedPreferenceHeaderPosition():int

2.4 Using the interface PreferenceFragmentListener

The interface PreferenceFragmentListener can be implemented by clases in order to be notified, when the currently shown preference fragment of a PreferenceActivity has been changed. Therefore the class PreferenceActivity provides the method addPreferenceFragmentListener(PreferenceFragmentListener):void, which allows to add such an listener to the activity. The method removePreferenceFragmentListener(PreferenceFragmentListener):void allows to remove a previously added listener. Listeners, which are implementing the class PreferenceFragmentListener must override the two methods onPreferenceFragmentShown(int,PreferenceHeader,Fragment):void and onPreferenceFragmentHidden():void. The first method is invoked when a preference header has been selected and its fragment is shown. It contains the preference header, its position and the fragment, which has been shown, as parameters. The second method is invoked, when a previously shown fragment has been hidden. This method is only called when the preference headers become shown on a device with a small display, which therefore does not support to show the preferences split-screen, or if all preference headers have been removed.

3 Manipulating the activity's appearance

The class PreferenceActivity and its layout files were designed in order to allow easy manipulation of its appearance. For this reason the class provides various getter methods, which allow to reference the child views, its layout consists of. To support the most important modifications of the child views appearances, the class even provides dedicated setter methods, which for example allow to change the width of the navigation as used on tablets. These getter and setter methods, as well as the possibility to take influence on the way, the preference headers are visualized, will be discussed in the following subdivisions of this section.

3.1 Referencing the activity's child views

Basically, the activtiy, which is provided by this library, consists of two fragments - the navigation and the fragment, which belongs to the currently selected preference header, if any. On smartphones only one of the fragment can be present at once, as they are shown fullscreen. On tablets both fragments are always present at the same time. Although the class PreferenceActivity does not provide any methods to directly access these fragments, the appearance of the layout they are included in, can be adapted by referencing the child views, which are listed in the following table. Because of the fact, that only one fragment can be present at once on smartphones, some of the getter methods shown below always return null when the library is run on such devices.

Getter method Explaination
getFrameLayout():FrameLayout Returns the frame layout, which contains the activity's views. It is the root view of the activity. Additional views, such like floating action buttons, can be added to the frame layout in order to hover all other views of the activity.
getPreferenceHeaderParentView():ViewGroup Returns the parent view of the fragment, which contains the navigation. On smartphones this parent view is also used to show the appropriate fragment, when a preference header is selected.
getPreferenceScreenParentView():ViewGroup Returns the parent view of the fragment, which contains the fragment of the currently selected preference header. On smartphones this method always returns null.
getPreferenceScreenContainer():ViewGroup Returns the view group, which contains the view, which can be referenced by the getPreferenceScreenParentView-method. This view is used to apply a horizontal margin. On smartphones this method always returns null.
getBreadCrumb():TextView Returns the view, which is used to display the title of the currently selected preference header. On smartphones or if the library's custom toolbar is used, this method always returns null, as the title is shown in the toolbar instead.

3.3 Taking influence on the PreferenceHeaderAdapter

The methods, which have been discussed in the previous two sections of this documentation do only allow to modify the appearance of the navigation and preference screen, but not of the preference headers itself. The adapter, which is used to visualize these preference headers can be retrieved by using the method getListAdapter():PreferenceHeaderAdapter, which returns an instance of the class PreferenceHeaderAdapter. This class offers methods, which allow take influence on how the preference headers are visualized.

The first possibility to change the appearance of the preference header is to change the selector, which is used to highlight the currently selected navigation item. The selector, which is used by default, is defined within the drawable resource R.drawable.selector_light. To use a custom drawable instead, the method setSelectorId(int):void can be used to pass the drawable's resource ID to the adapter. To retrieve the resource ID of the selector, which is currently used, the adapter offers the method getSelectorId():int. As an alternative to the default selector, which is meant to be used together with light themes, the drawable resource R.drawable.selector_light provides a selector for dark themes. In the last section of this documentation, the possibility to specify the selector and further attributes from within a theme, will be discussed.

Not only the selector, but even the layout, which is used to visualize each preference header, can be changed by passing the layout's resource ID to the method setViewId(int):void. If no custom layout is set, the drawable resource R.layout.preference_header_item is used instead. The resource ID of the currently used layout can be retrieved by using the method getViewId():int. The default layout contains an ImageView with the ID android.R.id.icon and two TextViews with the IDs android.R.id.title and android.R.id.summary. These views are used by the adapter to visualize the icon, title and summary of each preference header. Therefore, when using a custom layout, the layout definition should contain the same view types with identical IDs. If one of the views is missing, this will not cause the app to crash, but the affected properties of the preference headers will not be visualized in such case.

The class PreferenceHeaderAdapter implements the method getView(int, View, ViewGroup):View to create and return the views, which are used to visualize the single preference headers. If it is necessary to take influence on how these views are created, it is possible to register an instance of the type PreferenceHeaderDecorator at the adapter. This interface defines the method onApplyDecorator(int, PreferenceHeader, View, ViewHolder):void, which is called by the adapter whenever a specific preference header has to be visualized. The method's parameter contain the preference header, which should be visualized, as well as its position, the view, which has been created by the adapter and a view holder, which allows to performantly reference the view's child views. To add a PreferenceHeaderDecorator, which should be used to influence the creation of views, the method addDecorator(PreferenceHeaderDecorator):void of the class PreferenceHeaderAdapter can be used. The method removeDecorator(PreferenceHeaderDecorator):void allows to remove the decorator afterwards.

3.4 Changing the appearance of preferences

The library overrides the default appearance of preferences using the custom preference layout file /res/layout/preference.xml. Overriding the layout is only performed, when no custom layout has been set by using the preference's android:layout attribute or by using the corresponding method setLayoutResource(int):void. If an indent should be added to single preferences, for example in order to create a hierarchy when using the attribute android:dependency, like it is possible by using specifying the attribute android:layout="?android:attr/preferenceLayoutChild" when using the Android SKD's built-in PreferenceActivity, the attributy android:layout="@layout/preference_child" should be used instead. This will add an additional left padding to the preference, while still being visualy consistent with the default preference appearance. Alternatively it is of course also possible to use any other custom layout by specifying the attribute android:layout. The layout should contain an ImageView with the ID @android:id/icon and two EditText widgets with the IDs @android:id/title and @android:id/summary. The previously mentioned file /res/layout/preference.xml may be used as a template.

4 Changing the activity's behavior

This section is about the possibilities, which allow to take influence on the behavior of a PreferenceActivity. These possibilities concern the behavior of the toolbar's or action bar's back button, the option to hide the navigation and to initially select a specific preference header when the activity is started.

4.1 Overriding the navigation icon

The class PreferenceActivity overrides the behavior of its toolbar's or action bar's back button by default. This functionality, which can be turned on and off by using the method overrideNavigationIcon(boolean):void, does only take effect on smartphones. To retrieve, whether the behavior of the back button is currently overridden, the method isNavigationIconOverridden():boolean can be used. When this functionality is enabled, clicking the back button will cause the activity to resume to the navigation, whenever a preference header is currently selected. If the activity's toolbar or action bar does not show the back button by default, it will become visible while a preference header is selected. The intention of this functionality is to allow adapting the behavior of the toolbar's or action bar's back button to the bahavior of the device's back button, which acts exactly the same.

4.2 Hiding the navigation

Another possibility to take influence on the activity's behavior is to hide the navigation. This prevents the user to switch between the currently selected preference header and an other one. To hide or show the navigation, the method hideNavigation(boolean):void can be used. To retrieve, whether the navigation is currently hidden, the method isNavigationHidden():boolean is provided. Alternatively, a boolean extra can be passed via the intent, which is used to start the activity. The extra must contain the key PreferenceActivity.EXTRA_NO_HEADERS. Hiding the navigation espacially makes sense, when the acticity is launched with a specific preference header initially selected (see the folling section for further information).

4.3 Initially selecting a preference header

It is possible to use the intent, which is passed to the activity when it is started, to specify a preference header, which should be initially selected. This option allows to start the actvity in order to offer a specific group of preferences to the user. At the moment, it is only possible to initially select a preference header, which specifies a fragment. This decision has been made in order to preserve compatibility to the Android SKD's built-in PreferenceActivity, which offers the same functionality using the same intent extras, which will be discussed in the following.

The following table shows the intent extra, which allows to specify the preference header, which should be initially selected, by passing the full qualified class name of its fragment. Furthermore, the table contains the intent extras, which allow to specify alternative bread crumb titles and alternative parameters, which should be passed to the preference header's fragment as soon as it is shown.

Key Explaination Value type
PreferenceActivity.EXTRA_SHOW_FRAGMENT Allows to specify the preference header, which should be intially shown, by passing full qualified class name of its fragment.
String
PreferenceActivity.EXTRA_SHOW_FRAGMENT_ARGUMENTS Allows to pass alternative parameters to the fragment of the preference header, which should be initially shown.
Bundle
PreferenceActivity.EXTRA_SHOW_FRAGMENT_TITLE Allows to specify an alternative bread crumb title for the preference header, which should be initially shown.
CharSequence, String or string resource
PreferenceActivity.EXTRA_SHOW_FRAGMENT_SHORT_TITLE Allows to specify an alternative bread crumb short title for the preference header, which should be intially shown.
CharSequence, String or string resource

The alternative titles and parameters will be kept as long as the preference header remains selected, even if the orientation of the device changes or the activity is temporary brought to the background. But as soon as the user selects an other preference header, these properties will be lost and therefore the preference header's default properties are used if it will become re-selected afterwards. To prevent the user from selecting an other preference header, the navigation may be hidden. This possibility was already discussed in the previous section.

5.3 Using the interface WizardListener

Furthermore, it is possible to register listeners, which therefore must implement the interface WizardListener at a PreferenceActivity, which is used as a wizard. This interface defines methods, which are called when a wizard's next-, back- or finish-buttons are clicked by the user or when the user is about to finish the activity using the device's back button or the action bar's back button. All of these listener methods, which are listed below, must return an instance of the class Bundle, which allows to determine, whether a several action should be executed, or not. If the returned bundle is not null, the action will be executed, otherwise not. For example, this allows to verify the data the user has entered in the currently shown preference or to show a dialog, which asks for confirmation when the wizard is about to be skipped.

onNextStep(int, PreferenceHeader, Fragment):Bundle

onPreviousStep(int, PreferenceHeader, Fragment):Bundle

onFinish(int, PreferenceHeader, Fragment):Bundle

onSkip(int, PreferenceHeader, Fragment):Bundle

As shown above, all methods of the interface WizardListener define parameters, which contain the position of the currently selected preference header, the preference header itself and the fragment, which is currently shown. To add such a listener to an instance of the class PreferenceActivity, its method addWizardListener(WizardListener):void may be used. In case a listener, which has been registered this way, should not be notified on the wizard's events anymore, the method removeWizardListener(WizardListener):void can be used to remove the listener from the activity.

6 Using the class PreferenceFragment

Since version 1.1.0 the library provides the class PreferenceFragment, which is extended from the Android SDK's equally named class. This class may be used to show the preferences, which belong to a preference header. In order to achieve this, the full qualified class name of a class, which is extended from the class PreferenceFragment must be passed to a preference header by either using the appropriate setter method or by specifying it via XML. In addition to the functionality, which is provided by the Android SDK's built-in class, the class, which is provided by this library, offers the possibility to show a button at the bottom of the fragment, which may be used to restore the default values of the fragment's preferences.

6.1 How to show the restore defaults button

To show the button, which allows to restore the default values of the preferences, which are contained by an instance of the class PreferenceFragment, the classes' method showRestoreDefaultsButton(boolean):void may be used. To retrieve, whether the button is currently shown, the method isRestoreDefaultsButtonShown():boolean may be used instead. Instead of using the showRestoreDefaultsButton-method, the button may also be shown by passing an appropriate argument to the fragment. To specify, whether the button, which allows to restore the default values, should be shown, a boolean extra with the name PreferenceFragment.EXTRA_SHOW_RESTORE_DEFAULTS_BUTTON can be specified within the bundle, which is passed to fragment. To pass such an argument to a preference header's fragment, it is for example possible, to specify them as the preference header's extras. This is either possible by using the appropriate setter method of the class PreferenceHeader or by specifying the extras within an <intent>-tag, like shown in the "Examples" section of this documentation, if the preference header is defined via XML.

When the button, which allows to restore the default values of the fragment's preferences, is clicked by the user, the method restoreDefaults():void is implicitly called. This method can also be called manually in order to restore the default values.

6.2 Modifying the appearance of a PreferenceFragment

In order to be able to change the appearance of a PreferenceFragment, the views, which are contained by the fragment, can be referenced by using the following getter methods. Beware, that if one of the views is currently not shown, the corresponding getter method will return null.

Getter method Explaination
getFrameLayout():FrameLayout Returns the frame layout, which contains the fragment's views. It is the root view of the fragment. Additional views, such like floating action buttons, can be added to the frame layout in order to hover all other views of the fragment.
getRestoreDefaultsButton():Button Returns button, which allows restore the default values of the preferences, which are contained by the fragment. If the button is currently not shown, null will be returned instead.
getButtonBar():ViewGroup Returns the view group, which contains the button, which allows to restore the default values of the fragment's preferences. If the button is currently not shown, null will be returned instead.

The setter-methods can only be used when the button, which allows to restore the default values, is currently shown. 6.3 Using the interface RestoreDefaultsListener

Similar to the interface WizardListener, the interface RestoreDefaultsListener defines methods, which allow to supervise when the default values of a PreferenceFragment should be restored. Such a listener, which can be registered to the fragment using the method addRestoreDefaultsListener(RestoreDefaultsListener):void, must implement the following methods. If an already registered listener should be removed from the fragment in order to not be notified when the default values should be restored, anymore, the method removeRestoreDefaultsListener(RestoreDefaultsListener):void may be used.

onRestoreDefaultValuesRequested(PreferenceFragment):boolen

onRestoreDefaultValueRequested(PreferenceFragment, Fragment, Object):boolean

onRestoredDefaultValue(PreferenceFragment, Fragment, Object, Object):void

The onRestoreDefaultValuesRequested-method is called when the user clicks the button, which allows to restore the default valus of a PreferenceFragment. When the fragment's restoreDefaults-method is call programmatically instead, this listener method will not be called. The method's boolean return value allows to determine, whether restoring the default values should actually be executed, or not. This pattern e.g. allows to show a dialog, which asks the user for confirmation about the intented action. In case all registered listeners have returned true, for each preference, whose default value should be restored, the listener's onRestoreDefaultValueRequested-method is called. Implementing this method allows full control over which preferences' default values should be restored. When returning true, the default value of the preference will be restored, otherwise the preferences' current value will be kept. Furthermore, for each preference whose default value has been restored, the onRestoredDefaultValue-method is called in order to perform additional actions, which may be necessary because of the preference's value change.

7 Theme attributes

Like already mentioned at the beginning of this documentation, the theme, which is applied to a PreferenceActivity has to be derived from an AppCompat-theme, otherwise an exception will be thrown at runtime. The recommended themes to use are Theme.AppCompat.NoActionBar, if a dark theme should be used, or Theme.AppCompat.Light.NoActionBar, if a light theme should be used instead. When a theme, which includes an action bar, like for example Theme.AppCompat is used, the library's custom toolbar can not be used and a regular Holo-styled action bar will be shown instead.

When the library is used together with a theme, which does not include a Holo-styled action bar, a separate theme, which varies from the one used for the PreferenceActivity, can be applied on the activity's toolbar. As the toolbar's background color is taken from the theme attribute colorPrimary, its text color is taken from the attribute android:textColorPrimary and the color of its buttons is taken from the attribute android:textColorSecondary, this might be usefull, if different colors should be used for the toolbar and the activity's theme. The following theme attributes allow to specify a theme for the toolbar itself, as well as for the menu, which might be shown when the toolbar's menu is selected:

Attribute name Format Explaination
toolbarTheme
reference
 The theme, which should be applied on the toolbar itself
toolbarPopupTheme
reference
 The theme, which should be applied on the toolbar's popup menus

Unfortunately, the way how themes are defined and applied in Android does not allow to automatically adapt the activity's appearance depending on whether a light or dark theme is used. As the default appearance of the activity is intended to be used together with a light theme, the activity's appearance has to be adapted manually when using a dark theme.
